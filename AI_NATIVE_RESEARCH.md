# Making `sb-cli` AI‑Native: A Feature Blueprint for Agents and Humans (Mid‑2026)

## TL;DR
- **Build on what you already have, don't rebuild.** `sb-cli` (Rust, clap 4.5 derive, reqwest, serde_json, single static `sb` binary, TTY-aware `--format human|json`) is already most of the way to "AI-native." The highest-leverage additions are: (1) a `sb skills init` generator that writes SKILL.md/AGENTS.md/CLAUDE.md/Cursor/Copilot files, (2) an `sb mcp serve` subcommand using the official Rust MCP SDK (`rmcp`, now on its 1.x line), and (3) hardening the machine contract (stable exit codes, structured JSON error objects, `--dry-run`/`--yes`, `--fields`, machine-readable help).
- **Treat agents and humans as two audiences of one contract, not two products.** Every mutation should degrade gracefully: interactive confirmation for humans on a TTY, `exit code + JSON confirmation envelope + --yes` for agents. Keep data on stdout, diagnostics on stderr, and let `--format` auto-switch (which sb already does).
- **Favor local-first, no-telemetry approaches throughout.** The MCP server should default to stdio (local subprocess, no network), skills are just local Markdown files, and auth should stay in the OS keychain (which sb already supports via the `keyring` crate) plus env-var overrides for headless agents. Avoid any cloud-upload or telemetry — it's both against the user's stated preference and unnecessary for every feature listed here.

## Key Findings

**The industry has converged on a small set of conventions in 2026.** For agent instructions, `AGENTS.md` is the cross-tool baseline (an open standard donated to the Linux Foundation's Agentic AI Foundation in December 2025). Per OpenAI (openai.com/index/agentic-ai-foundation/, Dec 9, 2025), AGENTS.md, "since its release in August 2025… has been adopted by more than 60,000 open-source projects and agent frameworks including Amp, Codex, Cursor, Devin, Factory, Gemini CLI, Github Copilot, Jules and VS Code among others." For reusable capabilities, Anthropic's **Agent Skills** (`SKILL.md` with YAML frontmatter, released as an open standard) is supported across Claude Code, OpenAI Codex CLI, Gemini CLI, Cursor, and GitHub Copilot. For live tool access, **MCP** (Model Context Protocol) is the standard; per the Linux Foundation press release (Dec 9, 2025) it has "more than 10,000 published MCP servers now covering everything from developer tools to Fortune 500 deployments," adopted by "Claude, Cursor, Microsoft Copilot, Gemini, VS Code, ChatGPT and other popular AI platforms," with official SDKs in Rust (`rmcp`), Python, and TypeScript and stdio + Streamable HTTP as the two live transports (SSE deprecated).

**A strong counter-current says "for a local CLI, a well-designed CLI beats an MCP server."** Multiple practitioners (Arcjet, Stripe's link-cli, the "stcli" author, Firecrawl, the Trilogy AI essay) report deleting MCP servers in favor of CLIs because MCP tool schemas permanently consume context-window tokens, while a CLI is discoverable on demand via `--help`. The mature answer, which sb should follow, is: **make the CLI itself excellent for agents first, then offer an optional `mcp serve` as a thin wrapper over the same core.** Arcjet's write-up (blog.arcjet.com/designing-a-cli-for-ai-agents/) is the design north star: "The CLI is not a second product and neither is the MCP server," and both interfaces "expose the same underlying concepts… they should not invent separate domain models."

**Token efficiency is now a first-class design concern.** Agents pay per token and lose reasoning quality as context fills. Concretely: add `--fields` to trim large objects, paginate instead of dumping unbounded arrays, keep help text dense, and consider a compact tabular output mode (TOON/CSV) for uniform lists.

## Details

### A. Self-description & introspection (machine-facing)

**1. Machine-readable help / command manifest.** *What:* a way for an agent to learn the entire command surface in one call — e.g. `sb --help --format json` or a dedicated `sb schema` command emitting a JSON/JSON-Schema description of every command, subcommand, flag, type, default, and required-ness. *Why:* the first thing an agent does with an unfamiliar CLI is run `--help`; a structured version removes brittle text parsing and can double as the source of truth for MCP tool generation and shell completions. *Trend:* Google's `gws` CLI exposes `--schema` per command; Stripe's link-cli exposes `--schema`; the "rewrite your CLI for agents" guidance recommends a `--describe`/schema command. *sb mapping:* clap 4.5's derive API already holds the full `Command` graph in memory; you can walk `Command::get_subcommands()`/`get_arguments()` recursively and serialize to JSON with the `serde_json` you already depend on. This is a modest new `sb schema` command with no new dependencies.

**2. Rich, self-documenting `--help` on every command.** *What:* each command's help includes a one-line purpose, argument table with types/defaults, and 2–3 example invocations. *Why:* for agents, help text is the tool description, parameter spec, and usage guide in one; for humans it's progressive disclosure. *Trend:* clig.dev ("display extensive help text when asked"), and the DEV "CLI tools agents want to use" guide both stress example-rich help. *sb mapping:* clap derive supports doc-comment help and `after_help`/`long_about` for examples; sb's commands already have good descriptions — add an `Examples:` block to the high-traffic ones (`page create`, `daily`, `sync`, `query`).

**3. Shell completion as agent-discoverable schema.** *What:* completion metadata enumerates commands/flags/value-hints. *Why:* the same structure that powers tab-completion is a discovery affordance. *sb mapping:* sb already ships `sb completions <shell>` via `clap_complete` for bash/zsh/fish/elvish/powershell — this is done; because it's generated from the same clap graph it stays in sync automatically.

### B. Output modes (machine + human)

**4. Structured output with graceful TTY/pipe degradation.** *What:* human-readable tables when attached to a terminal, JSON when piped. *Why:* the single most important agent-compatibility feature; agents skip text parsing entirely. *Trend:* clig.dev's "human-readable output is paramount… have machine-readable output where it does not impact usability"; Gibil's `--json` pattern; GitHub CLI's `--json`. *sb mapping:* **already implemented** — sb's `--format human|json` "defaults to `human` when stdout is a TTY and `json` when piped." This is a genuine strength. Two refinements: (a) ensure **every** command (not just readers) honors `--format json`, including mutations returning a result object; (b) consider `--format jsonl` for streaming lists (`sb page list`, `sb daily` history, `sb logs`).

**5. Strict stdout/stderr separation.** *What:* data to stdout, all logs/progress/spinners/diagnostics to stderr. *Why:* lets `sb page list --format json | jq` never break because a spinner corrupted the stream; agents parse stdout, ignore stderr. *Trend:* clig.dev, Gibil ("stdout the agent parses; stderr the agent ignores"). *sb mapping:* sb uses `indicatif`/`console` for progress and `tracing`/`tracing-subscriber` for logs — verify progress bars and `--verbose` debug logging all go to stderr, leaving stdout pure. `--quiet` (already present) should suppress everything but the data payload.

**6. Token-efficient output: `--fields` and pagination.** *What:* a `--fields id,name,...` selector and enforced `--limit`/pagination on list commands. *Why:* a single large object or unbounded list can consume a big fraction of an agent's context; trimming fields cut round-trip token usage by ~60% in one practitioner's tests. *Trend:* Stripe's stcli ("Reduce by default. Every command has a --fields flag"); Google gws NDJSON `--page-all`; Anthropic's tool-response truncation guidance. *sb mapping:* `sb page list` already has `--limit`; add `--fields` to the JSON serializer (filter the serde_json `Value` before printing) and make `sb query`/`sb describe` respect it. This directly benefits the Runtime-API commands that can return large index results.

**7. Compact tabular format for uniform lists (optional).** *What:* an optional CSV/TOON output for large arrays of same-shaped objects. *Why:* TOON (Token-Oriented Object Notation) and CSV drop repeated keys. Per TOON's own benchmark (github.com/toon-format/toon), "TOON achieves 76.4% accuracy (vs JSON's 75.0%) while using 39.9% fewer tokens on these datasets," measured across Claude Haiku 4.5, Gemini 3 Flash, GPT-5 Nano and Grok-4.1-fast on 209 data-retrieval questions, with the authors citing 30–60% savings on large uniform arrays of objects compared to formatted JSON. *Caveat:* TOON is an emerging, non-standardized format and is worse for deeply nested data — treat as an opt-in for flat lists only, not a default. *sb mapping:* `sb daily` history and `sb page list` are exactly uniform-row data; a `--format csv` (or `toon`) would be a cheap, high-value token saver.

**8. Color control.** *What:* auto-disable color when not a TTY, honor `NO_COLOR` and a `--no-color` flag. *Why:* stray ANSI codes corrupt machine parsing; `NO_COLOR` is a widely honored convention. *sb mapping:* sb already has `--no-color`; ensure it also auto-detects non-TTY and reads the `NO_COLOR` env var (the `console` crate supports this).

### C. Determinism & safety

**9. Stable, documented exit codes.** *What:* a fixed taxonomy — e.g. 0 success, 1 general error, 2 auth error, 3 input/validation error, 4 confirmation-required, 5 not-found/conflict — documented and never reshuffled. *Why:* agents check exit codes before parsing output; distinct codes let them branch/retry without grepping stderr. *Trend:* Arcjet publishes a 0–4 code table; clig.dev; the Gibil and DEV guides. *sb mapping:* sb uses `thiserror` for error types — map each error variant to a stable `process::exit` code and document the table in `--help`/README. This is one of the highest-value, lowest-cost additions.

**10. Structured error objects in JSON mode.** *What:* when `--format json`, errors are emitted as JSON (on stderr) with `{ "error", "code", "remediation" }`. *Why:* gives agents two independent failure signals (exit code + parseable body) and a remediation string enables self-correction. *Trend:* Arcjet's exact pattern (`{"error","code","remediation"}`); the "fail loudly and specifically" skill-design pattern. *sb mapping:* extend the `thiserror` variants with a serializable representation; include remediation like "Run `sb auth set` or set SB_TOKEN."

**11. `--dry-run` everywhere it mutates.** *What:* preview changes without applying. *Why:* lets an agent (and human) validate a destructive/large operation safely. *Trend:* universal in agent-friendly CLIs (aider `--dry-run`, gws `--dry-run`). *sb mapping:* sb **already** has `--dry-run` on `sync`/`sync pull`/`sync push` — extend the same flag to `page delete`, `page move`, and `template new` so agents can rehearse edits to your notes.

**12. Non-interactive confirmation: `--yes`/`--force` + confirmation envelopes.** *What:* destructive ops prompt interactively on a TTY but, for agents, either require `--force`/`--yes` or return a "confirmation_required" JSON envelope with the exact command to re-run. *Why:* prevents agents from stalling on prompts they can't answer, while keeping a human approval gate for dangerous actions. *Trend:* Arcjet's exit-code-4 + `confirmCommand` envelope is the gold standard; deployhq's "add --yes/--force for autonomous agents." *sb mapping:* `sb page delete` already has `--force`; generalize to a consistent `--yes` across all mutations, and consider Arcjet-style JSON confirmation envelopes for `page delete`/`page move` when `--format json` and no `--yes`.

**13. Idempotency and clear create/upsert semantics.** *What:* document which operations are idempotent; offer upsert-style behavior where sensible (sb's template `openIfExists: true` is a good example). *Why:* agents retry on failure; idempotent operations make retries safe. *sb mapping:* `page create` on an existing page should have well-defined behavior (error with exit code 5, or `--upsert`); document it.

**14. Sandboxing / permission posture for `sb shell`.** *What:* keep genuinely dangerous capabilities off by default. *Why:* an agent driving the CLI shouldn't be one hallucination away from arbitrary server command execution. *sb mapping:* `sb shell <cmd>` (execute a command on the server) is **already opt-in/disabled by default** — keep it that way, and in any generated SKILL/AGENTS files explicitly warn agents that it is destructive and gated.

### D. Interactive vs. headless operation

**15. Robust TTY detection + `--no-input`.** *What:* auto-disable all prompts, spinners, and `$EDITOR` launches when stdin/stdout isn't a terminal, plus an explicit override flag. *Why:* the #1 way agent-friendly CLIs break is blocking on an interactive prompt. *Trend:* Speakeasy's retrofit story ("Interactive prompts blocked execution… we added flags that let agents opt out"); ICLR 2026 AMBIG-SWE finding that agents "default to non-interactive behavior." *sb mapping:* sb already skips `$EDITOR` "automatically when not attached to a terminal" and falls back from fzf to numbered prompts — extend this so the interactive fzf/numbered picker is fully bypassable (require the name argument or error with a helpful code) under `--no-input`/non-TTY.

**16. Full env-var + config-file configuration.** *What:* every setting reachable via env var and config file, with a documented precedence order. *Why:* agents configure via environment, not interactive setup. *sb mapping:* **already excellent** — sb documents precedence (CLI flags > env vars > keychain > `.sb/config.toml` > defaults) and states "Every config field has a `SB_` environment variable." This is a model implementation; just surface it in the generated AGENTS.md so agents know to use `SB_SERVER_URL`/`SB_TOKEN`.

### E. Agent-facing documentation

**17. `sb skills init` — generate agent instruction/skill files (the user's priority #1).** *What:* a generator subcommand that writes agent-directory files describing how to drive sb. Cover these formats/locations (all current as of 2026):
  - **`AGENTS.md`** (repo root): the cross-tool open standard (Linux Foundation Agentic AI Foundation, Dec 2025). Plain Markdown, no schema. Read natively by Claude Code, Codex CLI, Cursor, Copilot, Gemini CLI, aider, Windsurf/Devin, Amazon Q. Keep it under ~200–500 lines, imperative, command-first. This is the single most important file to generate.
  - **`SKILL.md`** as `.claude/skills/<name>/SKILL.md` (and portable to `~/.claude/skills`, `~/.codex/skills`, Cursor, Copilot): YAML frontmatter with required `name` (lowercase/hyphens, ≤64 chars, must match folder) and `description` (≤1024 chars, states *what* and *when*), Markdown body, optional `scripts/`, `references/`, `assets/` subdirs. Progressive disclosure: name+description (~100 tokens) load at startup, body on activation, references on demand. Avoid angle brackets in frontmatter (injection risk). A sb skill would bundle recipe commands ("to append to today's journal: `sb daily <text>`").
  - **`CLAUDE.md`** (project root + `~/.claude/`): Claude Code's memory file; can `@import` other files. Generate a short one that points to AGENTS.md.
  - **`.cursor/rules/*.mdc`**: Cursor's format — YAML frontmatter (`description`, `globs`, `alwaysApply`) + Markdown; four activation modes (always / auto-attach via globs / agent-decided / manual). Cursor-locked; generate only if targeting Cursor. Note `.cursorrules` (root single-file) is now deprecated but still read.
  - **`.github/copilot-instructions.md`** (+ `.github/instructions/*.instructions.md` with `applyTo` glob frontmatter): GitHub Copilot.
  - **`.windsurf/rules/*.md`** (now `.devin/rules/` preferred post-rebrand): single `trigger` frontmatter field; hard char limits (6,000 global / 12,000 per file).
  *Why:* lets sb "teach" any coding agent how to use itself, correctly and consistently, without the user hand-writing five files. *Trend:* this is exactly what tools like rule-porter, Skillwright, and Stripe Projects ("writes coding agent skills into the local project directory") do. *sb mapping:* a new `sb skills init [--target claude|cursor|copilot|windsurf|agents|all]` that renders templates (embed them in the binary) populated from sb's own command schema (see #1). Because sb already emits its full command graph via clap, the AGENTS.md/SKILL.md bodies can be generated from the source of truth rather than hand-maintained. **Caveat to heed:** research (Gloaguen et al., 2026) found LLM-*generated* context files can *reduce* agent performance by over-instructing — so generate concise, command-focused files (commands, exit-code table, env vars, safety notes), not verbose prose.

**18. `llms.txt` generation (optional).** *What:* the `/llms.txt` convention (Jeremy Howard/Answer.AI, Sept 2024) is a Markdown manifest of an H1, a blockquote summary, and curated links — designed for *websites*. *Why/caveat:* it's a web-inference-time asset, not a CLI-native one, and adoption remains niche with no major LLM vendor officially confirming they consume it at crawl time. For sb, the CLI-native equivalents (AGENTS.md, machine-readable help) matter far more. *Recommendation:* deprioritize; `sb skills init` could optionally emit an `llms.txt` for a project's notes space, but it's low value unless sb ever ships a docs site.

**19. Man pages / embedded examples.** *What:* generate man pages and keep examples in help. *sb mapping:* `clap_mangen` (sibling to the `clap_complete` you already use) can generate man pages from the same command graph — cheap to add.

### F. MCP server from the CLI (the user's priority #2)

**20. `sb mcp serve` using `rmcp` (official Rust MCP SDK).** *What:* a subcommand that runs sb as an MCP server, exposing its commands as MCP tools. *Why:* lets MCP-native clients (Claude Desktop, Cursor, VS Code, ChatGPT) call sb without shell access; some environments prefer typed tool calls over shell strings. *Trend:* Stripe link-cli (`npx @stripe/link-cli --mcp` for stdio, `serve` for HTTP), Codex's `codex mcp-server`, Google gws `gws mcp`. The pattern is a `mcp serve` subcommand over the *same core* as the CLI ("one source of truth, two interfaces"). *sb mapping specifics:*
  - **SDK:** `rmcp` (the official Rust SDK, now on its **1.x** line — latest published release 1.8.0 per docs.rs; note the repo's "Migrating to 1.x? See the migration guide for breaking changes"). Add with `features = ["server", "transport-io", "macros"]`. It fits sb's existing `tokio` async runtime and `serde`/`schemars` derive style. Tools are defined with `#[tool_router]`/`#[tool(description=...)]` macros and `Parameters<T>` structs that derive `JsonSchema` — very close to how clap derive already models sb's args.
  - **Transport:** default to **stdio** (local subprocess, zero network, no telemetry — matches the user's privacy preference). Offer Streamable HTTP only behind an explicit flag for multi-client/remote use; SSE is deprecated, don't implement it. **Critical stdio gotcha:** never write to stdout in an stdio MCP server (it corrupts JSON-RPC) — sb must route all `tracing` logs to stderr in this mode.
  - **Primitives:** map read commands (`page read`, `page list`, `query`, `describe`, `server ping`) to **tools** with `readOnlyHint: true`; map mutations (`page create/edit/delete/move/append`, `daily`, `sync`) to tools with `readOnlyHint: false` and `destructiveHint: true` on delete/move. Set `idempotentHint`/`openWorldHint` accurately (defaults are pessimistic: unannotated tools are assumed destructive and open-world). These annotations let clients auto-approve safe reads and gate destructive writes — and Claude Code uses `readOnlyHint` to run read tools in parallel. Consider exposing sb's page content as MCP **resources** (URI per page) and canned queries as **prompts**.
  - **Auto-generation:** because sb's command surface is already a structured clap graph, you can generate MCP tool definitions from the same source used for help/skills — avoid hand-maintaining a parallel tool list.
  - **Token discipline:** don't expose 20+ tools if a handful of outcome-oriented tools suffice; large tool catalogs bloat every client's context (200–500 tokens per tool schema, always loaded). Reuse `--fields`/pagination in tool outputs.

**21. MCP discovery/registration (optional, later).** *What:* publish a `server.json` to the official MCP Registry (metadata-only, namespace-verified via GitHub OAuth as `io.github.cam-barts/sb`), and/or a `.well-known/mcp.json` server card (SEP-1649) if ever hosted over HTTP. *Why:* makes the server discoverable by aggregators (Smithery, Glama, VS Code's `@mcp`). *Caveat:* only meaningful once `sb mcp serve` exists and is stable; the registry is still preview/v0.x, discovery SEPs are proposed-not-merged, and Claude Code's `/mcp` doesn't browse the registry. Deprioritize until the server ships.

### G. Auth & secrets for agent contexts

**22. Env-var token override for headless auth.** *What:* `SB_TOKEN`/`--token` take precedence so an agent can authenticate without a keychain prompt or browser. *Why:* agents can't do interactive/browser OAuth; env injection is the only path that works headless. *Trend:* Arcjet (`ARCJET_TOKEN` beats keychain), gws (`GOOGLE_WORKSPACE_CLI_TOKEN`), the WorkOS CLI-auth guide. *sb mapping:* **already implemented** — sb's precedence puts `--token` and `SB_TOKEN` above keychain/config. Keep it, and document it in the generated AGENTS.md.

**23. Keychain by default for humans; never echo secrets.** *What:* store tokens in the OS keychain; mask by default; require an explicit `--reveal` to show them. *Why:* keeps plaintext secrets off disk (a real risk now that coding agents read `.env` files opportunistically). *sb mapping:* **already implemented** — sb uses the `keyring` crate (Apple/Windows native + secret-service) gated behind `auth.keychain=true`, and `sb config show` masks tokens unless `--reveal`. This aligns with the "inject at runtime, never write plaintext, keep secrets out of agent-readable files" guidance. One addition: ensure secret values never appear in `--verbose`/`tracing` logs or JSON error objects.

### H. Packaging & distribution for agent usability

**24. Single static binary + broad, reproducible install paths.** *What:* one dependency-free binary, installable via multiple channels, with version pinning. *Why:* agents (and CI) need deterministic, no-runtime installs; `uvx`/`npx`-style ephemeral runs and pinned versions make agent environments reproducible. *sb mapping:* sb is **already** a single static binary across 5 platforms via GitHub Releases + `cargo install --git`, with `sb version` reporting commit/build-date/target (great for agents to log exact version). Concrete upgrades:
  - **Add `[package.metadata.binstall]` to Cargo.toml.** The subagent confirmed this section is **currently absent**. Adding it enables `cargo binstall sb-cli` to fetch your prebuilt release tarballs instead of compiling — much faster for agent/CI setup. (Note: cargo-binstall's quickinstall *fallback* path has opt-out telemetry; pointing binstall at your own GitHub-release-hosted binaries avoids that entirely.)
  - **Provide a `curl … | sh` install script** (the common agent-friendly install path) and an `sb upgrade`/self-update subcommand.
  - Consider publishing to crates.io so `cargo install sb-cli --version x.y.z` gives pinned, deterministic installs.

### I. Observability (local-first, no telemetry)

**25. Local structured logging via `--verbose`, no cloud upload.** *What:* debug/trace logging to stderr, opt-in, structured. *Why:* agents and humans need to see what happened for self-correction, without any data leaving the machine. *sb mapping:* sb already uses `tracing`/`tracing-subscriber` with `env-filter` — expose `RUST_LOG`/`SB_LOG` and keep all of it on stderr and strictly local. **Explicitly do not add usage analytics, phone-home, or crash-upload** — this honors the user's stated no-telemetry preference and avoids the negative example set by tools like the Stripe CLI (which "includes a telemetry feature that collects usage data") and cargo-quickinstall's default stats collection. If you ever want metrics, prefer a fully local, opt-in log the user owns.

### J. Reference implementations worth studying
- **Arcjet CLI** — the definitive write-up on dual agent/human CLI design (exit-code taxonomy, JSON error objects with remediation, confirmation envelopes, shared core between CLI and MCP).
- **Stripe `link-cli`** — `--schema` per command, `--mcp` stdio + `serve` HTTP, agent polling with `--interval/--max-attempts`, non-TTY defaults to compact output.
- **Google Workspace `gws`** (Rust, ~23.9k stars) — schema introspection, JSON-default output, `mcp` subcommand and 100+ bundled skills, all generated from one source of truth.
- **GitHub CLI `gh`** — `--json` + `--jq` + `--template` as the canonical structured-output pattern.
- **aider** — every flag has an env var, `--yes`/`--yes-always`, `--dry-run`, `CONVENTIONS.md` read-only context files, fully scriptable.
- **stcli** — extreme token-minimization (JSON-only, errors to stderr, `--fields`, a skill file of runnable recipes).

## Recommendations

**Stage 1 — Harden the machine contract (days, no new deps).** These make sb agent-safe with the crates you already have:
1. Define and document a **stable exit-code table** (map `thiserror` variants → codes). *Benchmark to proceed:* an agent can distinguish auth vs. not-found vs. validation failures without reading stderr.
2. Emit **structured JSON error objects** (`error`/`code`/`remediation`) under `--format json`; send them to stderr.
3. Confirm **stdout=data / stderr=diagnostics** everywhere; verify `indicatif` spinners and `tracing` logs never touch stdout.
4. Extend **`--dry-run`** to `page delete/move` and `template new`; generalize **`--yes`** across all mutations.
5. Add **`--fields`** and confirm `--limit`/pagination on all list/query commands.
6. Ensure **`--no-input`/non-TTY** fully bypasses fzf/numbered pickers and `$EDITOR`.

**Stage 2 — Add discoverability & skills (the user's priority #1).**
7. Add **`sb schema`** (JSON dump of the clap command graph) — the source of truth for the next steps.
8. Add **`sb skills init --target all`** generating `AGENTS.md` + `SKILL.md` (+ CLAUDE.md/Cursor/Copilot/Windsurf on request), populated from `sb schema`. Keep files concise and command-first (heed the Gloaguen et al. over-instruction finding). *Benchmark:* drop the generated AGENTS.md into a Claude Code/Cursor session and confirm the agent correctly runs `sb daily`, `sb page create`, and respects the exit-code/`--yes` conventions.
9. Add **`clap_mangen`** man-page generation (trivial, same graph).

**Stage 3 — MCP server (the user's priority #2).**
10. Add **`sb mcp serve`** with `rmcp` 1.x (stdio default, HTTP behind a flag), exposing a *small* set of outcome-oriented tools generated from `sb schema`, with accurate `readOnlyHint`/`destructiveHint` annotations, all logs forced to stderr. *Benchmark to proceed to registry:* Claude Desktop/Cursor can list and call sb tools, reads auto-approve, deletes prompt for confirmation.
11. Optionally expose pages as MCP **resources** and saved queries as **prompts**.

**Stage 4 — Distribution & polish.**
12. Add **`[package.metadata.binstall]`**, a `curl | sh` installer, and an `sb upgrade` self-update.
13. Optionally publish to **crates.io** for pinned installs; later, publish `server.json` to the **MCP Registry** once `mcp serve` is stable.

**What would change these recommendations:** If you find agents rarely use `sb mcp serve` (because they have shell access and prefer the CLI directly — the dominant 2026 pattern), stop investing in MCP breadth and instead double down on Stage 1–2 (the CLI contract + skills). If agents frequently blow their context on sb output, prioritize `--fields`/TOON/pagination sooner. If you never ship a docs site, skip `llms.txt` entirely.

## Caveats
- **Some cited figures are single-source or emerging.** TOON's token-savings claims come from the TOON project's own benchmark (76.4% vs 75.0% accuracy at 39.9% fewer tokens on their 209-question data-retrieval set); an independent arXiv benchmark by Ivan Matveev (arXiv:2603.03306, submitted Feb 8, 2026) found the opposite for *generation*: "plain JSON generation shows the best one-shot and final accuracy," and for simple structures constrained-decoding JSON's low token usage "outperformed even TOON." Treat TOON as an opt-in optimization for uniform tabular data, not a default.
- **The "MCP vs CLI" debate is unsettled.** Prominent practitioners argue a good CLI beats an MCP server for local, single-user tools due to MCP's context-window tax; others find MCP wins for remote/unfamiliar APIs. For sb (local, single-user notes tool), the CLI-first + optional-MCP ordering above is the defensible call, but reasonable engineers disagree.
- **Agent-file standards are moving fast.** `.cursorrules` is deprecated in favor of `.cursor/rules/*.mdc`; Windsurf rebranded to Devin with `.devin/rules/` now preferred; Claude Code merged custom slash commands into skills (`.claude/skills/`). Any generator must be versioned and easy to update as these drift. `AGENTS.md` is the safest single bet.
- **MCP Registry and `.well-known/mcp.json` are pre-1.0.** The official registry is in preview (v0.x), discovery SEPs (SEP-1649/1960) are proposed-not-merged, and major clients don't yet auto-consume the registry. Don't over-invest in discovery infrastructure yet. (The `rmcp` SDK itself, by contrast, has reached a stable 1.x line.)

- **LLM-generated context files can backfire.** Research (Gloaguen et al., 2026) found auto-generated agent instruction files can reduce performance and inflate cost by over-broadening exploration. Generate lean, specific, command-focused files and let users edit them.
